# Design: Wind field

## Purpose

Replace the single, uniform, manually-entered wind with a spatially-,
altitude-, and time-resolved **wind field** fetched from the meteo-map-server.
The field feeds three consumers: the map's vector barb layer, the route tab's
ground-speed/ETE computation, and the side-view wind projection. Manual wind
remains the offline fallback.

## The wind.json contract (cross-repo seam)

Served at `<serverUrl>/wind.json`, generated by `meteo-map-server`'s
`generate_wind_json.py`:

```json
{
  "reference_time": "2026-06-15T12:00:00Z",
  "times": ["2026-06-15T12:00:00Z", "..."],
  "levels_ft": [0, 3300, 6500, 10000],
  "grid": [
    { "lat": 48.0, "lon": 7.5,
      "u": [[u_t0_l0, u_t0_l1, ...], ...],   // [time][level], m/s
      "v": [[...], ...] }
  ]
}
```

**Conventions that MUST stay in sync across both repos:**
- `u`/`v` are wind components in **m/s**, meteorological convention (u eastward,
  v northward). "Direction from" = `atan2(-u, -v)`.
- `levels_ft` ascending; on the server they map from pressure levels
  (`HPA_TO_FT`: 1000→0, 900→3300, 800→6500, 700→10000 ft).
- `times` ascending; inner `u`/`v` arrays are indexed `[time][level]`.
- Grid is a regular lat/lon mesh; only fully-populated points are emitted.

## Client architecture

```
wind.json ──► WindFieldProvider (C++ singleton)
                 parse + cache(disk) + O(1) cell index
                 windAt(lat,lon,altFt,time)  ── trilinear interp
                 windAtStep(...)             ── nearest time step (no time interp)
                    │
       ┌────────────┼─────────────────────────────┐
       ▼            ▼                              ▼
  FlightMap.qml   FlightRoute (route tab)     SideviewQuickItem
  vector barbs    Leg::integrate / legNav     windProfile (projected)
                  waypointETAs
```

### WindFieldProvider (`src/weather/WindFieldProvider.{h,cpp}`)

- **Interpolation**: `windAt` is **trilinear** — bilinear in lat/lon, linear in
  altitude, linear in time (each clamped to range). `windAtStep` snaps time to the
  nearest forecast step (no time interpolation) — used by the map.
- **Performance**: a quantized `(lat,lon)→index` hash (`m_index`, built in
  `parse()`) makes corner lookup **O(1)**. Critical: a barb at every grid point is
  O(n) lookups, so a linear search was O(n²).
- **Caching**: writes `wind.json` to `CacheLocation/meteo_france/` and loads it on
  construction → available offline / at restart.
- **Staleness**: `isStale()` is true once "now" passes the last forecast step;
  `isUsable() = hasData && !isStale`. Consumers fall back to manual wind when not
  usable.
- `windFromUV(uKn, vKn)` converts components → `Weather::Wind` (speed + dirFrom).

### Route integration (`src/navigation/Leg.{h,cpp}`, `FlightRoute.{h,cpp}`)

- `Leg::integrate(wfp, manualWind, aircraft, startTime, startAltFt, endAltFt)`
  steps along the great circle (4–24 steps by length); per step the altitude ramps
  linearly between the leg's waypoint altitudes, the wind is sampled in 4-D with
  the clock advancing, and the local GS comes from the wind triangle.
  **ETE = ∫ ds/GS**. Returns `{ete, gs, wind}`.
- `FlightRoute::legNav(index, …)` walks from `departure` accumulating each leg's
  ETE to get that leg's start time, then integrates it → `{ete, gs, wind, fuel}`
  for the navlog.
- `FlightRoute::waypointETAs(…)` returns the arrival time at every waypoint (same
  accumulation) — used by the side view for ETA-aware sampling.
- `Navigator` owns `departureTime` (trip start) and `cruiseAltitudeFt`; the
  weather time slider drives `departureTime`.

### Map vector layer (`FlightMap.qml`)

- A `MapItemView` over `WindFieldProvider.gridPoints` draws a barb per point via a
  small `Canvas`, sampled with `windAtStep` at `windAltitudeFt` and the
  slider-selected time. Thinned by zoom; interpolation skipped for hidden points.
- `GlobalSettings.showWindLayer` is the single toggle for map + side view.

## Why interpolation lives where it does

- **Map barbs** sit on grid nodes → no spatial interpolation needed; time is
  snapped to the nearest step. Effectively native data.
- **Route + side view** sample arbitrary lat/lon, altitude, and time → full
  trilinear interpolation is required and correct.

## Testing

High-value, deterministic, no UI/network. Prefer **invariants** over magic numbers.

**WindFieldProvider** (parse a hand-built wind.json):
- Node lookup: `windAt` at an exact grid node + exact level + exact step returns
  that node's stored value.
- Bilinear: midpoint between four equal-corner nodes returns that value; a known
  asymmetric set returns the analytic average.
- Altitude/time interpolation hit known midpoints; clamping outside range.
- `windAtStep` returns the nearest step's value with no blending.
- `windFromUV`: `u=-10,v=0 → from 270°, 10 kt`; `u=0,v=-10 → from 360°`.
- Staleness flips when now crosses the last `times` entry.
- O(1) index returns the same result as a brute-force corner scan (guards the
  optimisation).

**Leg::integrate** (invariants):
- Zero wind ⇒ `ETE = dist/TAS`, `GS = TAS`.
- Pure headwind W ⇒ `GS = TAS − W`; tailwind ⇒ `TAS + W`; `GS ≤ TAS + W` always.
- Altitude ramp: a leg climbing through two levels with different winds yields a
  GS between the two single-level results.

**FlightRoute**:
- `waypointETAs` strictly increasing; `etas[0] == departure`; final ETA −
  departure == Σ leg ETEs.
- `legNav` ETE matches `integrate` for that leg's start time.

**Cross-repo contract** (the most important guard):
- Commit one canonical `wind.json`. Python test: `generate_wind_json` over a fixed
  synthetic dataset reproduces that schema (keys, ordering, units). C++ test:
  `WindFieldProvider` parses it and returns expected winds at chosen points. If
  either side changes the schema or the u/v convention, the other's test fails.

**Golden snapshot**:
- Fixed route + fixed wind.json → expected per-leg ETE/GS within tolerance; catches
  subtle numeric regressions a pure-invariant test would miss.
